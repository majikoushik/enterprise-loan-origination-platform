# Operational Runbook

This guide is intended for developers and operations engineers to diagnose and monitor the Enterprise Loan Origination Platform.

## 1. Checking Service Health
All backend microservices expose standard ASP.NET Core health check endpoints.

- **Liveness Check**: `GET /health/live`  
  Returns `Healthy` if the application process is running. Used for container orchestration (Kubernetes/Azure Container Apps).
- **Readiness Check**: `GET /health/ready`  
  Returns `Healthy` only if the application is ready to accept traffic and all internal dependencies (e.g., SQL Server) are reachable.
- **Deep Health Check**: `GET /health`  
  Returns full health status.

To view the health of all services simultaneously, visit the **System Health Dashboard** in the Angular frontend:  
`http://localhost:4200/system-health`

## 2. Using Correlation ID for Troubleshooting
Every HTTP request initiated by the frontend or external clients is tagged with an `X-Correlation-ID` header.
- The Correlation ID is automatically generated by an Angular interceptor.
- It propagates down to the API Gateway, Backend API, background workers, and Audit/Notification logs.
- When searching logs in the terminal or Azure Log Analytics, query by `CorrelationId` to trace the full path of a business transaction.

*Example query in Log Analytics:*
```kusto
AppTraces
| where Properties["CorrelationId"] == "123e4567-e89b-12d3-a456-426614174000"
| order by TimeGenerated asc
```

## 3. Where Logs Go in Azure
- **Locally**: Serilog writes all structured logs to the console window using a standard format:  
  `[Timestamp] [Level] [ServiceName] [CorrelationId] Message`
- **Azure Production**: The platform is wired to send logs via the `Serilog.Sinks.ApplicationInsights` package. Logs, Request Telemetry, and Exceptions are pushed to Azure Application Insights and can be queried via Azure Log Analytics.

## 4. Common Failure Scenarios
### Scenario A: Database Connection Refused
**Symptoms**: `/health/ready` returns `Unhealthy`. Serilog logs show `SqlException`.  
**Triage**: Check if `sqlserver` docker container is running. Verify `ConnectionStrings` in `appsettings.json`.

### Scenario B: Notification Simulation Fails
**Symptoms**: Application status gets stuck on submitted, but email/SMS isn't logged in `Audit.Api`.  
**Triage**: Ensure `Notification.Worker` is running. Check its console logs for exceptions fetching data from Service Bus / HTTP Webhook.

## 5. Safe Logging Guidelines
- **Never log PII**: Passwords, SSN, Credit Card Numbers.
- **Do not leak stack traces**: Unhandled exceptions are masked by `GlobalExceptionHandler` returning `ProblemDetails` to the client, while logging the full exception safely in Application Insights.

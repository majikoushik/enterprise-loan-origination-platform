Implement Epic 6: Audit Logging based on AGENTS.md and the existing repository.

Build the next business feature for the `enterprise-loan-origination-platform`: Centralized Audit Logging and Business Event Traceability.

Before implementation:

1. Read `AGENTS.md`.
2. Review the current repository structure.
3. Review the implementation from Epic 0, Epic 1, Epic 2, Epic 3, Epic 4, and Epic 5.
4. Review existing Customer, Loan Application, Eligibility, Status Tracking, and Notification implementations.
5. Review existing Angular routes, services, components, tests, documentation, CI workflow, Docker setup, and event abstractions.

Epic 6 Goal:

Add centralized audit logging for important business actions so that the loan origination journey becomes traceable, reviewable, and enterprise-ready.

This epic should demonstrate architecture maturity around governance, operational traceability, compliance-readiness, event history, and supportability.

Backend scope:

1. Implement centralized Audit Logging in `Audit.Api` or the existing audit service structure.
2. Create a clean domain model for `AuditEvent`.
3. Add request and response DTOs if the audit service exposes APIs.
4. Add persistence using Entity Framework Core.
5. Add database migration support.
6. Create audit records for important business events across the platform.
7. Add APIs to query audit history.
8. Add structured logging for audit event creation and query actions.
9. Use correlation ID support if available.
10. Ensure audit events include safe, minimal, non-sensitive business context.
11. Add Swagger/OpenAPI documentation readiness.
12. Add unit tests and integration tests where practical.

* Customer registered

Use consistent API response and error response patterns according to `AGENTS.md`.

Audit event design expectations:

1. Audit events should represent business actions that already happened.
2. Audit events should be immutable after creation.
3. Audit event names should be business-friendly.
4. Audit metadata should not contain secrets, passwords, full financial payloads, or unnecessary personal data.
5. Audit events should include correlation ID wherever available.
6. Audit events should include source service name.
7. Audit query APIs should support traceability by entity ID, customer ID, and correlation ID.

Example audit event names:

* `CustomerRegistered`
* `LoanApplicationSubmitted`
* `EligibilityCheckCompleted`
* `LoanApplicationStatusChanged`
* `NotificationRequested`
* `NotificationDeliveryCompleted`
* `NotificationDeliveryFailed`
